# RAUD-20 Read-Endpoint Abuse Controls

- Prior status: Not Fixed. `GET /assistant/api/suggest` and `GET /assistant/api/faq` were public read paths with deterministic success/degrade behavior but no explicit abuse bounds. Local pre-change probes on March 10, 2026 showed ordinary `200` JSON responses for both endpoints and repeated same-query probes still returning `25 x 200`, confirming there was no read-path throttling in place.
- Post-change status: Fixed
- Verification level tag: VC-UNIT + VC-PURE + targeted functional + local runtime

## Evidence

### Pre-change baseline

Before editing:

- `curl -k -sS -D '<headers>' -o '<body>' "https://ilas-pantheon.ddev.site/assistant/api/suggest?q=housing&type=all"` returned `HTTP/2 200` with top-level key `suggestions`.
- `curl -k -sS -D '<headers>' -o '<body>' "https://ilas-pantheon.ddev.site/assistant/api/faq?q=eviction"` returned `HTTP/2 200` with top-level keys `results`, `count`.
- Repeated same-query probes to each endpoint still returned `25 x 200`, so there was no explicit bounded abuse behavior on either GET read path before the remediation.
- Existing targeted functional baseline also passed before editing:
  - `testSuggestEndpointAccessible`: `1 test`, `10 assertions`
  - `testFaqEndpointAccessible`: `1 test`, `10 assertions`

### Post-change implementation proof

Implemented:

- `web/modules/custom/ilas_site_assistant/src/Service/AssistantReadEndpointGuard.php` adds endpoint-specific per-IP Flood API protection for `suggest` and `faq`, keyed by `RequestTrustInspector`-resolved client IP rather than raw `Request::getClientIp()`.
- `web/modules/custom/ilas_site_assistant/src/Controller/AssistantApiController.php` now invokes the guard on controller-executed read requests, returns deterministic `429` JSON with `Retry-After` and endpoint-shaped empty payloads on throttle, and preserves existing `200`/`404` degraded behavior for dependency failures.
- `web/modules/custom/ilas_site_assistant/config/install/ilas_site_assistant.settings.yml`, `config/ilas_site_assistant.settings.yml`, and `web/modules/custom/ilas_site_assistant/config/schema/ilas_site_assistant.schema.yml` now define explicit `read_endpoint_rate_limits` defaults for `suggest` (`120/min`, `1200/hour`) and `faq` (`60/min`, `600/hour`).
- The runtime/test documentation now records the new public contract: normal `200` read responses are unchanged, while throttled responses are explicit `429` with `Retry-After`, `type = rate_limit`, `request_id`, and endpoint-shaped bodies.

## Residual risks / unknowns

- Identical cache-hit GETs can still be served by Drupal’s existing read cache before the controller-level throttle executes. The remediation preserves that cache behavior intentionally, and the functional throttle proof uses varied queries to exercise the controller path directly.
- This change does not add edge/CDN/WAF controls, anomaly dashboards, CAPTCHA/challenge flows, or admin metrics for read throttles.
- Pantheon/runtime proof is still local-only until the updated module code and config are deployed to `dev`/`test`/`live`.

## Rollback notes

- Roll back the new read-endpoint guard service, controller/config wiring, targeted tests, doc updates, and this runtime artifact as one code-only bundle.
- No schema migration, content rewrite, or database rollback is required.
- The `function_exists()` guard added around `ilas_site_assistant_get_canonical_urls()` in `ilas_site_assistant.module` should be rolled back with the test/bootstrap changes only if the wider suite no longer needs controller-level function stubbing.

## Still-unverified surfaces

- Long-duration distributed abuse patterns beyond the per-IP controller-level bounds added here
