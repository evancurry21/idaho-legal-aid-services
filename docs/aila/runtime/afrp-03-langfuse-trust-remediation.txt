AFRP-03 Langfuse Trust Reproof
Date: 2026-03-19
Verification level tag: Partially Fixed

Prior Langfuse trust posture
- Fresh HTTP `207`, queue depth, and stored config were available, but those
  were transport signals, not trustworthy observability proof.
- Local direct and queued synthetic probes had been shown previously, but the
  trust gap remained between "accepted by ingestion API" and "visible as a
  current, privacy-safe trace."
- Real assistant request-path proof was the weakest surface: a fresh
  `/assistant/api/message` `request_id` did not reliably resolve account-side
  during earlier verification attempts.
- Queue-loss behavior was only implicit. Operators could see queue depth/age,
  but not explicit counters for max-depth drops, stale discards, invalid-shape
  discards, or retryable suspensions.
- Pantheon `dev` / `test` / `live` still exposed the older `ilas:langfuse-status`
  surface, so hosted proof depended on ad hoc probe output and account-side
  checks rather than a first-class lookup command.

Post-change posture
- Repo/runtime now exposes sanitized account-side proof through
  `ilas:langfuse-lookup <trace_id>`.
- Local `ilas:langfuse-status` now exposes explicit export outcomes:
  `drop_max_depth`, `discard_invalid_shape`, `discard_missing_enqueued_at`,
  `discard_stale`, `discard_disabled`, `discard_missing_credentials`,
  `discard_non_retryable_http`, `send_success`, `send_partial_207`,
  `retryable_suspend`.
- Local direct, queued, and real request-path traces are all freshly re-proven
  account-side after the AFRP-03 changes.
- Current hosted Pantheon `dev` / `test` / `live` direct and queued probes are
  freshly re-proven account-side, and one hosted `live` request-path trace is
  freshly re-proven account-side with the expected vector/request fields.
- Queue-loss behavior is now explicitly accounted for in local runtime proof via
  max-depth and stale-item drills.
- No privacy-boundary expansion was introduced. Visible Langfuse `input` and
  `output` remain summary strings only, and the exported metadata remains
  limited to hashes, buckets, booleans, counts, source classes, and reason
  codes.

Request-to-export lifecycle summary
- `AssistantApiController::message()`
  - starts trace with `request_id` as trace ID
  - ends trace with minimized output summary and request-path metadata
- `LangfuseTracer`
  - buffers trace/span/generation/event payloads
  - finalizes a single `trace-create` plus observation batch
- `LangfuseTerminateSubscriber`
  - flushes once per request on response/terminate
  - enforces `langfuse.max_queue_depth`
  - stamps `enqueued_at`
- Drupal queue `ilas_langfuse_export`
- `LangfuseExportWorker`
  - validates shape
  - enforces `langfuse.max_item_age_seconds`
  - posts to `POST /api/public/ingestion`
  - records explicit outcome counters
- Langfuse SaaS
  - trace detail: `GET /api/public/traces/{trace_id}`
  - recent-trace fallback: `GET /api/public/traces?limit=25`

Trust-gap conclusion
- HTTP `207` is necessary but not sufficient.
- Queue depth moving `0 -> 1 -> 0` is necessary but not sufficient.
- Trustworthy Langfuse signal now requires:
  - current runtime truth/status
  - successful queue drain where applicable
  - account-side trace proof for the exact trace ID
- AFRP-03 now provides that proof path locally, and current hosted releases are
  freshly re-proven account-side on `dev` / `test` / `live`.

Custom exporter decision
- Keep the custom exporter.
- Reasoning:
  - it preserves the metadata-only privacy contract already enforced in
    `LangfusePayloadContract`
  - it matches Drupal queue semantics directly
  - AFRP-03 now adds the missing trust surfaces around lookup and queue-loss
    accounting without replacing the exporter
  - inspection of `dropsolid/langfuse-php-sdk` did not produce a proven
    reduction in the observed trust gaps and did not provide a clear
    environment-field advantage over the custom path

Residual risks / unknowns
- No Pantheon deploy was performed in this prompt, so hosted operator-visible
  AFRP-03 surfaces (`ilas:langfuse-lookup`, export counters in
  `ilas:langfuse-status`) are still deployment-gated even though current hosted
  traces themselves are proven account-side.

  read minimized metadata `environment` / `env` keys for the true runtime
  environment until that surface is improved.
- Trace visibility can lag materially behind ingestion acceptance; the lookup
  command needed a recent-trace fallback and longer retry window to avoid false
  negatives on fresh traces.
- `VC-PURE` remains red on unrelated workflow/vector-merge failures outside
  AFRP-03 scope.
