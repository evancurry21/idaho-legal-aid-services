1. Prior FAQ retrieval state
----------------------------

Before the AFRP-01 code change:

- Missing lexical FAQ index wiring was not the cause of the sampled bug.
  `faq_accordion` was configured and enabled in `local`, `dev`, `test`, and
  `live`.
- English `eviction` FAQ probes on 2026-03-18 were environment-split:
  - `dev`: English-only parent URLs
  - `test`: English-only parent URLs
  - `local`: mixed-language leak (`/resources/...`, `/es/...`, `/sw/...`,
    `/nl/...`)
  - `live`: mixed-language leak (`/resources/...`, `/es/...`, `/sw/...`,
    `/nl/...`)
- `GET /assistant/api/faq` browse/category mode leaked mixed-language labels on
  `local` and `live`, including `Preguntas frecuentes`, `Maswali Yanayoulizwa
  Mara kwa Mara`, and `Veelgestelde vragen`.
- Root-cause proof from sampled paragraphs:
  - `pid 343`: paragraph language `en`, parent node language `en`,
    parent URL `/resources/evictions`
  - `pid 641`: paragraph language `en`, parent node language `es`,
    parent URL `/es/resources/desalojos`
  - `pid 904`: paragraph language `en`, parent node language `sw`,
    parent URL `/sw/resources/kufukuzwa`
  - `pid 1167`: paragraph language `en`, parent node language `nl`,
    parent URL `/nl/resources/uitzettingen`
- Conclusion from the baseline: paragraph-level `search_api_language=en` was
  insufficient for FAQ locale isolation because translated parent nodes still
  produced translated parent URLs and translated category scopes.

2. Post-change state
--------------------

Repo change summary:

- `FaqIndex` now applies one shared same-language gate before returning FAQ
  results from:
  - lexical Search API search
  - vector supplementation
  - legacy fallback search
  - `getById()`
  - category aggregation (`getCategories()`, `getCategoriesLegacy()`)
- The gate validates resolved parent scope, not just paragraph language:
  - parent URL must match the current request language
  - parent node language must match when available
  - unresolved parent scope fails closed
- `/assistant/api/faq` response shapes stay unchanged.
- `/assistant/api/message` inherits the same behavior because the controller
  routes FAQ retrieval through `FaqIndex::search()`.

Local post-change runtime state:

- `FaqIndex->search("eviction", 5)` now returns only English parent URLs:
  - `faq_343` -> `/resources/evictions`
  - `faq_346` -> `/resources/evictions`
- `GET /assistant/api/faq?q=eviction` now returns only English FAQ results in
  the sampled local runtime:
  - `faq_343` -> `/resources/evictions`
- `GET /assistant/api/faq` category mode now returns only English labels in the
  sampled local runtime:
  - `Legal Advice Line`
  - `Resources`
  - `FAQ`
  - `Senior Legal Risk Detector`
- `GET /assistant/api/faq?id=faq_641` now returns HTTP `404` with body
  `{"error":"FAQ not found"}` in local English runtime.

Hosted current-runtime state after the repo change:

- No Pantheon deploy occurred in this prompt, so hosted post-change behavior is
  still deployment-gated.
- Refreshed read-only probes confirm the current hosted runtime split still
  matches the pre-change baseline:
  - `dev`: English-only `FaqIndex->search("eviction", 5)` result set
  - `test`: English-only `FaqIndex->search("eviction", 5)` result set
  - `live`: mixed-language `FaqIndex->search("eviction", 5)` result set with
    `/es`, `/sw`, and `/nl` parent URLs still present
  - `live`: mixed-language category labels still present
