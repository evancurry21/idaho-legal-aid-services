AFRP-02 Runtime Truth Remediation
Date: 2026-03-18
Verification level tag: repo remediated / hosted expansion deployment pending

Prior runtime-truth posture
- `drush ilas:runtime-truth` existed, but the deployed helper and admin report were still observability-heavy.
- Operators could safely prove Langfuse, Sentry, Pinecone, GA, and a few runtime site settings, but they could not safely distinguish stored config from effective runtime behavior for assistant rate limits, conversation-logging invariants, retrieval runtime-only URL status, LLM readiness, or Voyage readiness.
- Multiple runbook sections still used `config:get ilas_site_assistant.settings ...` as if it were runtime proof for assistant settings. That was storage-only and misleading.
- Current divergence already existed before edits:
  - `local`, `dev`, and `test` reported stored `vector_search.enabled=false` with effective `true`.
  - `local`, `dev`, `test`, and `live` reported stored `langfuse.enabled=false` with effective `true`.
  - `live` reported stored `google_tag_id=false` with effective `true`.

Post-change posture
- `RuntimeTruthSnapshotBuilder` now preserves the top-level contract and expands the nested assistant-setting families:
  - message rate limits
  - session bootstrap thresholds
  - read-endpoint rate limits
  - conversation-logging resolved invariants
  - LLM provider/model/runtime-ready/cost-control/credential-presence
  - retrieval IDs plus runtime-only LegalServer URL status
  - Voyage runtime-ready/API-key/toggle state
  - prior Langfuse, Sentry, Pinecone, GA, diagnostics-token, and live-debug surfaces
- The admin report now exposes a broader `Runtime Truth Status` section with grouped tables for:
  - core assistant controls
  - retrieval and runtime-only URL status
  - observability and integration status
  - stored-vs-effective divergences
- The runbook now treats `ilas:runtime-truth` as the canonical assistant settings runtime truth path. `config:get` remains storage-only and historical for override-prone assistant settings.
