AFRP-14 — Language Isolation Across Fallback Retrieval Paths
=============================================================
Date: 2026-03-25
Prerequisite: AFRP-01 (FAQ language isolation), AFRP-13

1. Prior Fallback Language-Isolation State
------------------------------------------

Three ResourceFinder legacy retrieval paths lacked language filtering:

  a) findByTypeLegacy() — ResourceFinder.php:1636
     Triggered when Search API is unavailable. Calls loadLegacyResourceCandidates()
     which runs an entity query on node type 'resource' WITHOUT langcode condition.
     Results returned WITHOUT urlMatchesCurrentLanguage() post-filter.
     Risk: Cross-language resource results when Search API is down.

  b) findByTopic() — ResourceFinder.php:1743
     Public method always using loadLegacyResourceCandidates(). No language filter.
     Also called from findByTypeSearchApi() (line 826) for sparse-result topic boost,
     meaning unfiltered foreign-language results could enter an otherwise
     language-safe lexical Search API flow.
     Risk: Cross-language resources in topic-based lookups and topic boost.

  c) findByServiceArea() — ResourceFinder.php:1786
     Public method always using loadLegacyResourceCandidates(). No language filter.
     Risk: Cross-language resources in service-area lookups.

Root cause: loadLegacyResourceCandidates() (line 1834) queried nodes without
a langcode condition and returned built resources via buildIndexedResource()
without URL-based language validation.

All FaqIndex paths were already safe (AFRP-01):
  - Lexical: search_api_language + filterItemsByCurrentLanguage()
  - Vector: search_api_language + itemMatchesCurrentLanguage()
  - Legacy: filterItemsByCurrentLanguage() (line 1530)
  - Categories: filterItemsByCurrentLanguage() (line 1751)

ResourceFinder lexical and vector Search API paths were already safe:
  - Lexical: search_api_language condition (line 768)
  - Vector: urlMatchesCurrentLanguage() post-filter (line 1316)


2. Post-Change State
--------------------

Defense-in-depth fix applied (mirrors FaqIndex AFRP-01 pattern):

  Layer 1 — Query-level: Added ->condition('langcode', $this->getCurrentLanguage())
  to the entity query in loadLegacyResourceCandidates(). Prevents loading
  wrong-language nodes at the database level.

  Layer 2 — Post-filter: Added filterResourcesByCurrentLanguage() method and
  applied it in findByTypeLegacy(), findByTopic(), and findByServiceArea()
  after loadLegacyResourceCandidates() returns. Validates resolved URLs
  match the current language prefix (catches Drupal translation edge cases).

7. Residual Risks / Unknowns
-----------------------------

  a) Pinecone vector namespace: Vector index language partitioning is not verified
     at the index configuration level. Currently covered by urlMatchesCurrentLanguage()
     post-filter on vector results, but query-level namespace isolation is unverified.

  b) Cache key does not include language: buildQueryCacheKey() in ResourceFinder uses
     PiiRedactor::redactForStorage() on the query but does not include the current
     language code. If the same query hash is cached for EN, an ES request could
     receive EN results from cache. Pre-existing; outside AFRP-14 scope.

  c) Drupal entity query langcode edge cases: If a resource node exists only in ES
     (no EN translation), the langcode=en entity query condition should exclude it.
     The URL post-filter is the safety net for any Drupal translation fallback quirks.


8. Rollback Notes
-----------------

  Revert the three post-filter lines and the langcode condition in
  loadLegacyResourceCandidates(). filterResourcesByCurrentLanguage() can remain
  as dead code or be removed. No schema changes or config changes to reverse.
