# TOVR-07 Internal Telemetry Operationalization

- Timestamp (UTC): `2026-03-17T00:53:58Z`
- Verification level tag: `implementation-backed / partial-runtime`
- Scope: private health/metrics machine access, telemetry-map classification,
  privacy/retention continuity, and explicit separation of machine-consumable
  versus Drupal-only observability surfaces.

## Prior status

- Anonymous local `GET /assistant/api/health` returned HTTP `403` with JSON
  `error_code=access_denied`.
- Anonymous local `GET /assistant/api/metrics` returned HTTP `403` with JSON
  `error_code=access_denied`.
- `/assistant/api/health` and `/assistant/api/metrics` were route-gated only by
  `view ilas site assistant reports`, so external monitors without a Drupal
  session could not consume the existing SLO/cron/queue/metrics payloads.
- `/admin/reports/ilas-assistant`,
  `/admin/reports/ilas-assistant/conversations`, and conversation detail were
  already Drupal-admin/manual-review surfaces.

## Post-change status

- `/assistant/api/health` and `/assistant/api/metrics` now allow either:
  - Drupal permission `view ilas site assistant reports`, or
  - `X-ILAS-Observability-Key: <token>` matching runtime-only
    `ILAS_ASSISTANT_DIAGNOSTICS_TOKEN`.
- Anonymous/sessionless callers still receive controlled JSON
  `error_code=access_denied`; no detailed diagnostics became public.
- Health/metrics payload semantics were preserved:
  - health still returns `status`, `timestamp`, `checks`
  - metrics still returns `timestamp`, `metrics`, `thresholds`, `cron`, `queue`
  - metrics still includes `all_endpoints`, `by_endpoint`, `by_outcome`,
    `source_governance`, `vector_index_hygiene`, `session_bootstrap`,
    `cost_control`
- `/admin/reports/ilas-assistant` and conversation views remain Drupal-only.
- Metadata-only retention boundaries remain unchanged for
  `ilas_site_assistant_stats`, `ilas_site_assistant_no_answer`, and
  `ilas_site_assistant_conversations`.

## Signals that can currently drive alerts

- `SloAlertService` logger warnings for availability, latency, error rate, cron,
  and queue violations.
- Private machine-auth synthetic checks against `/assistant/api/health`.
- Private machine-auth synthetic checks against `/assistant/api/metrics`,
  including threshold and nested snapshot regressions.

## Signals that still require manual Drupal review

- Admin report dashboard trends in summary stats, top topics/destinations,
  content-gap hashes, quality signals, feedback, and review loop.
- Conversation list/detail review for opt-in QA/debugging.
- Direct DB inspection of analytics/no-answer/conversation tables.

## Residual risks and unknowns

- Pantheon positive-path HTTP verification is still outstanding because the
  runtime-only diagnostics secret could not be safely reproduced from CLI in
  this session even though safe boolean checks reported it present.
- Long-run cron cadence and queue drain behavior under sustained load remain
  unproven.
- Broad `VC-UNIT` and `VC-KERNEL` suites currently have unrelated failures,
  which adds suite noise around otherwise-passing TOVR-07 targeted coverage.
- Admin report and conversation surfaces remain manual/Drupal-only by design;
  TOVR-07 does not attempt to make those views public or anonymously machine
  readable.

## Rollback notes

- Restore `_permission: 'view ilas site assistant reports'` on
  `ilas_site_assistant.api.health` and `ilas_site_assistant.api.metrics`.
- Remove `AssistantDiagnosticsAccessCheck` service registration.
- Remove `ilas_assistant_diagnostics_token` runtime setting wiring from
  `settings.php`.
- Revert the corresponding unit/functional tests and documentation updates.
- Clear Drupal caches after rollback.

## Still-unverified surfaces

- Positive machine-auth HTTP checks against deployed Pantheon `dev`/`test`/`live`
  using the real web-runtime diagnostics secret.
- Alert-routing proof that an external monitor is actively consuming the new
  machine-auth path in each environment.
- Sustained cron/queue behavior under non-trivial load or over a longer time
