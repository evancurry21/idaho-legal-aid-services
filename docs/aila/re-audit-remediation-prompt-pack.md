# Re-Audit Remediation Prompt Pack

## How to use
1. Run prompts in the listed order unless a prompt explicitly allows parallel work.
2. Treat the current re-audit open findings as the closure target; do not inherit closure from older docs, TODOs, or prior prompt output.
3. Reconstruct the pre-change state before editing code, config, or docs.
4. Use the status taxonomy and verification tags in this pack exactly as written.
5. Return an evidence-backed report for every prompt; doc-only proof is never enough to mark a technical finding `Fixed`.

## Global constraints
1. Scope is limited to the current open re-audit set plus the final closure sweep.
2. Preserve `llm.enabled=false` in `live`; no prompt in this pack authorizes live LLM enablement.
3. No broad rewrite, no platform-wide refactor, and no unrelated UI redesign.
4. Pantheon verification is read-only for this pack.
5. Add or strengthen tests for every behavioral fix unless technically impossible; if proof is unavailable, return `Unverified` or `Partially Fixed`.
6. String-anchor tests, documentation guards, and config exports cannot close a runtime or security finding on their own.
7. Every prompt report must include prior status, post-change status, verification level tag, evidence, changed files, command-output summary, residual risks, rollback notes, and still-unverified surfaces.

## Audit contract
- Finding status taxonomy: `Fixed`, `Partially Fixed`, `Not Fixed`, `Regressed`, `Unverified`.
- Verification level tags: `Verified by execution`, `Verified by code inspection`, `Claim-only/doc evidence`, `Cannot verify`.
- Prior Findings Matrix schema: `Finding ID`, `Original issue summary`, `Category`, `Current status`, `Evidence`, `What changed`, `Remaining gap`, `Recommended next action`, `Severity now`.
- Closure rule: a finding is only `Fixed` when the mitigation exists in the correct layer and the prompt returns code-level proof plus meaningful test or runtime verification.
- Failure rule: if runtime or environment proof is unavailable, the prompt must return `Unverified`, not `Fixed`.

### Promptfoo Gate Integrity Addendum (2026-03-08)
1. Keep blocking thresholds unchanged unless the prompt proves the rubric itself is defective.
2. Treat TLS, auth, pacing, endpoint reachability, and 429 failures as harness issues until the prompt proves a product defect instead.
3. Preserve promptfoo failure adjudication evidence whenever failures are reclassified as harness, rubric, or product issues.
4. Do not accept simulated success in place of a real paced run against the target endpoint.
