# Production Hardening Follow-On Prompt Pack

## How to use
1. Run prompts in the listed order unless a prompt explicitly allows parallel work.
2. Treat this pack as a separate hardening wave; it does not silently reopen or close `RAUD-*` findings.
3. Reconstruct the current runtime state before editing code, config, or docs.
4. Require executed proof for every closure claim; doc-only or config-only proof is never enough.
5. For `PHARD-01` and `PHARD-02`, live operational proof is mandatory. If SaaS-side proof cannot be produced, return `Unverified`.
6. Use sanitized, non-PII synthetic traffic only for live verification.
7. Return an evidence-backed report for every prompt.

## Global constraints
1. Preserve `llm.enabled=false` in `live`; no prompt in this pack authorizes live LLM enablement.
2. Drupal-managed ILAS content remains the primary knowledge source.
3. Retrieval must remain taxonomy-aware and lexical-first; semantic supplements are gated behind proof and may not silently become the uncontrolled primary path.
4. Feedback capture in this pack means a small user-facing helpful/not-helpful signal plus reporting/dashboard integration, not a broad new support platform.
5. Do not expand transcript retention or observability payloads with raw user text.
6. No broad UI redesign, no net-new public channel, and no unrelated platform refactor.
7. Pantheon verification may use sanitized live requests, but avoid destructive writes and avoid PII.

## Audit contract
- Finding status taxonomy: `Fixed`, `Partially Fixed`, `Not Fixed`, `Regressed`, `Unverified`.
- Verification level tags: `Verified by execution`, `Verified by code inspection`, `Claim-only/doc evidence`, `Cannot verify`.
- Prior Findings Matrix schema: `Finding ID`, `Original issue summary`, `Category`, `Current status`, `Evidence`, `What changed`, `Remaining gap`, `Recommended next action`, `Severity now`.
- Closure rule: a finding is only `Fixed` when the mitigation exists in the correct layer and the prompt returns code-level proof plus meaningful test or runtime verification.
- Failure rule: if runtime, Pantheon, or SaaS proof is unavailable, the prompt must return `Unverified`, not `Fixed`.
- `PHARD-01` and `PHARD-02` special rule: live event/trace evidence, alert-route evidence, and named ownership evidence are all mandatory. Config presence, unit tests, or watchdog lines alone cannot close them.
- `PHARD-03..07` special rule: scenario coverage must include the exact plain-language cases in this pack. A generic test pass without those scenarios is insufficient.
- `PHARD-08` special rule: do not claim privacy closure from documentation or UI notice changes alone; runtime retention, access, and minimized payload behavior must also be proven.
