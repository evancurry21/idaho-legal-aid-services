# Tooling, Observability, and Vector Remediation Prompt Pack

## How to use
1. Run prompts in the listed order unless a prompt explicitly allows parallel work.
2. Treat the current working tree, current configuration, current CI/CD wiring, and current runtime behavior as the source of truth.
3. Use prior audit reports, backlog notes, and earlier markdown only to identify candidate surfaces to inspect; do not inherit their conclusions, scores, or recommendations.
4. Resolve every listed unknown before marking a prompt `Fixed` or `Complete`.
5. Use sanitized synthetic traffic for live checks and avoid printing secrets in command output.
6. Return an evidence-backed report for every prompt; doc-only proof is never enough.

## Global constraints
1. Scope is limited to Aila-specific tooling, observability, evaluation, release quality, runtime verification, and Pinecone/vector-search enablement.
2. Preserve `llm.enabled=false` in `live`; no prompt in this pack authorizes live LLM response generation.
3. Pinecone/vector search remains lexical-first and supplemental unless a prompt proves, implements, and verifies a controlled change.
4. No prompt may print, commit, or export secrets; verify secret wiring with presence checks, masked values, or account-side proof instead.
5. Pantheon work is read-only until the Pinecone enablement prompts explicitly authorize environment changes for `dev`, `test`, or `live`.
6. Add or strengthen tests for every behavioral or configuration change unless technically impossible; if proof is unavailable, return `Unverified` or `Partially Fixed`.
7. Every prompt report must include prior status, post-change status, verification level tag, evidence, changed files or `no code changes`, command-output summary, residual risks, rollback notes, and still-unverified surfaces.

## Fresh-review contract
1. Current code, current config, current environment overrides, current CI workflows, and current runtime behavior are authoritative.
2. Prior docs may identify surfaces to inspect but may not supply closure claims.
3. Unknowns use this certainty taxonomy: `confirmed`, `likely`, `possible`, `unproven`.
4. A prompt cannot be marked `Fixed` or `Complete` while any listed unknown remains `unproven`.
5. If account-side or environment-side proof is unavailable, return `Unverified`, not `Fixed`.

## Audit contract
- Finding status taxonomy: `Fixed`, `Partially Fixed`, `Not Fixed`, `Regressed`, `Unverified`.
- Verification level tags: `Verified by execution`, `Verified by code inspection`, `Claim-only/doc evidence`, `Cannot verify`.
- Tool inventory status taxonomy: `confirmed active`, `confirmed partial`, `installed but not used`, `configured but not proven active`, `stale/dead/incomplete`, `suspected only`, `not present`.
- Confidence taxonomy: `high`, `medium`, `low`.
- Closure rule: a prompt is only `Fixed` when the mitigation exists in the correct layer and the prompt returns code-level proof plus meaningful runtime, CI, or test verification.
- Failure rule: if runtime, Pantheon, CI, or SaaS proof is unavailable, the prompt must return `Unverified`.

### Pinecone activation addendum
1. Do not enable Pinecone/vector search in `live` until `dev` and `test` prove secret/config parity, index integrity, retrieval quality, observability coverage, and rollback safety.
2. Do not use Pinecone enablement to mask lexical retrieval defects that should be fixed in lexical normalization, taxonomy, or ranking logic.
3. Do not assume index freshness, collection contents, or account-side readiness from exported YAML alone.
4. Keep live enablement reversible with a clearly identified rollback switch and post-enable verification window.
