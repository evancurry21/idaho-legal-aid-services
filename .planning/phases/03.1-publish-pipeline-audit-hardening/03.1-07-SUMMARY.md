# Phase 03.1 Plan 07: PIPE-06 Dual-Remote Drift Detection Consolidation Summary

**One-liner:** Extended `describe_remote_status` with a typed 4th-column recovery command; updated all entry-point consumers (sync-check.sh, pre-commit-master-sync.sh) to emit it; replaced the Plan 01 stub with a real 4-case harness that exercises each drift type against /tmp repos.

## What Was Built

### Recovery-Command Table (status/remote → command)

| Status | Remote | Recovery Command |
|--------|--------|-----------------|
| `remote-ahead` | `github` | `npm run git:sync-master` |
| `remote-ahead` | `origin` | `npm run git:reconcile-origin` |
| `diverged` | `github` | `git log --left-right --cherry-pick --oneline github/master...master; npm run git:sync-master` |
| `diverged` | `origin` | `git log --left-right --cherry-pick --oneline origin/master...master; npm run git:reconcile-origin` |
| `missing` | `*` | `git remote add <name> <url>; git fetch <name>` |
| `in-sync` | `*` | (empty — no recovery needed) |
| `local-ahead` | `*` | (empty — expected push direction) |

### Task 1 — scripts/git/common.sh: 4th column added to describe_remote_status

The function now emits a 4-column tab-separated record:
```
status<TAB>remote_only<TAB>local_only<TAB>recovery_cmd
```

Recovery is computed via a `case "$status:$remote"` dispatch immediately before the final `printf`. Backward-compat: existing 3-column consumers (`IFS=$'\t' read -r STATUS C2 C3`) silently drop col4 — no parse error, no behavior change.

### Task 2 — scripts/git/sync-check.sh: thin consumer of 4th column

The `IFS=$'\t' read` now captures `recovery_cmd` as the 4th field. The hardcoded `Run: npm run git:sync-master` / `Run: npm run git:reconcile-origin` lines are replaced by:
```bash
if [[ -n "$recovery_cmd" ]]; then err "Recovery: $recovery_cmd"; fi
```
The `terminus env:code-log` inspect hint (origin-only) and the `git log --left-right` inspect line are preserved.

### Task 3 — scripts/ci/pre-commit-master-sync.sh: 4th column consumed

Both `IFS=$'\t' read` calls (github and origin) now capture a 4th column (`GITHUB_RECOVERY`, `ORIGIN_RECOVERY`). Each case branch emits:
```bash
[[ -n "$GITHUB_RECOVERY" ]] && err "Recovery: $GITHUB_RECOVERY"
```
All existing diagnostic `err` messages are preserved verbatim. The `git branch backup/recovery-<timestamp>` advisory in the diverged branch remains. The hook exits correctly on the current repo state (local-ahead of both remotes).

### Task 4 — tests/ci/dual-remote-invariant.sh: real 4-case harness

Replaces the Plan 01 stub (exit 64). Four cases run against ephemeral `/tmp` repos:

| Case | Setup | Expected Status | Expected Recovery Pattern |
|------|-------|-----------------|--------------------------|
| A | Local has unpushed commit | `local-ahead` | empty |
| B | github remote got new commit; local fetched | `remote-ahead` | `npm run git:sync-master` |
| C | origin remote got new commit; local fetched | `remote-ahead` | `npm run git:reconcile-origin` |
| D | github got commit AND local got different commit | `diverged` | contains `git log --left-right` + `git:sync-master` |

### Plan-Literal Choices

- **pre-push-strict.sh not updated:** The plan's must_have states "Existing consumers (pre-push-strict.sh:156, sync-check.sh:79, pre-commit-master-sync.sh:27) continue to parse the first 3 columns via `IFS=$'\t' read -r STATUS C2 C3`" — verifying backward compat, not mandating an update. Recovery hints at pre-push are redundant (pre-commit blocked first). Left intentionally as 3-col reader.

- **SC2034 pre-existing warning:** `DEFAULT_REMOTES` in common.sh triggers SC2034 (used externally by consumers that source the file). Pre-existing before this plan; not introduced by these changes. Verified by stashing the change and confirming the warning existed already.

- **Harness uses `init_repos()` (in-process) instead of command-substitution:** The plan suggested `local_repo="$(setup_repos ...)"` pattern. That approach set `TMPDIR_BASE` inside a subshell which doesn't propagate to the parent (needed for trap-on-EXIT cleanup). Fixed by making `init_repos` set `CASE_TMPDIR` directly in the parent scope — trap-on-EXIT cleanup works correctly.
